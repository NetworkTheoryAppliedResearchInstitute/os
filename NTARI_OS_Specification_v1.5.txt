## 1. Executive Summary

### 1.1 Purpose

NTARI OS is a network-first operating system based on Alpine Linux and ROS2
middleware. It serves cooperatives, mutual aid networks, community mesh
deployments, Teen Tech Centers, and rural ISPs — organizations that need
reliable, auditable, community-maintainable networking infrastructure.

v1.5 represents a major architectural evolution from v1.0–v1.4, which focused
on SoHoLINK as a federated AAA platform. v1.5 formalizes the shift to a full
**network-first operating system** where the ROS2 DDS communication graph is
the foundational abstraction. Every service runs as a node within that graph.
Local computation is a node's contribution to the network, not the machine's
primary purpose.

The system enables:

- **Cooperative network infrastructure** owned and governed by participants
- **Graph-native network visibility** via the Globe interface
- **Composable services layer** — deployments include only needed services
- **Community governance** of services, policies, and federation
- **Cooperative federation** via WireGuard + DDS domain bridging

### 1.2 Design Principles

1. **Minimal Footprint** — <100MB base (500MB with ROS2); deployable on
   single-board computers, repurposed routers, and low-power mesh nodes
2. **Offline-First** — Full functionality without internet connectivity
3. **Security by Default** — Hardened configuration, minimal attack surface,
   small auditable base
4. **Network-First Interface** — The ROS2 computation graph is the primary
   interface; no traditional desktop environment in the base system
5. **Multi-Architecture** — x86_64 and ARM64 (Raspberry Pi 4)
6. **Federation-Ready** — WireGuard tunnels + DDS domain bridging for
   inter-cooperative federation
7. **Community Governance** — Services layer is modular, community-replaceable,
   and AGPL-3.0 licensed throughout

### 1.3 Target Hardware

**Reference Platform**: Raspberry Pi 4 (4GB RAM), x86_64 mini-PC (8GB RAM)
**Supported Architectures**: x86_64, ARM64

**Minimum Requirements (Full Stack):**
- CPU: 2 cores, 1.5 GHz
- RAM: 512 MB (base only) / 1 GB (with ROS2) / 2 GB (full services)
- Storage: 4 GB (base) / 8 GB (with ROS2) / 32 GB (full services)
- Network: Ethernet or WiFi

**Deployment Variants:**

1. **Minimal Edge Node** — Base + Middleware only; graph participant, sensor/data
   publisher; ~400 MB disk, ~150 MB RAM idle
2. **Community Hub Node** — Full four-layer stack; DNS, DHCP, NTP, identity,
   file sharing, globe interface; ~500 MB disk, ~350 MB RAM idle
3. **Federation Gateway** — Base + Middleware + VPN node; WireGuard tunnel
   management between cooperatives
4. **Headless Server Node** — Base + Middleware + selected services; no globe
   interface; administered via SSH and ros2 CLI

### 1.4 Key Architectural Changes from v1.4

| Area | v1.4 | v1.5 |
|---|---|---|
| Base OS | Alpine Linux 3.19 | Alpine Linux 3.23 |
| Middleware | DDS abstraction (conceptual) | ROS2 Jazzy + Cyclone DDS (specified) |
| Interface | CLI/TUI admin dashboard | Globe network graph (prototype delivered) |

| Services | SoHoLINK (AAA focus) | Full services layer (15+ node packages) |

## 3. Base Layer

### 3.1 Alpine Linux 3.23

**Version**: 3.23 (current stable with SEQSENSE ROS2 APK support)
**Init**: OpenRC — simpler, more auditable, easier to customize for
non-standard boot sequences than systemd.
**C library**: musl libc — strict POSIX compliance, smaller attack surface
**Userland**: BusyBox — minimal, auditable
**Package manager**: APK from three repositories:
1. Alpine official packages
2. NTARI APK repository (services layer node packages)
3. SEQSENSE ROS experimental aports (ROS2 packages for Alpine)

### 3.2 Boot Priority

OpenRC starts services in this order:

```
1. Kernel + networking drivers
2. Network interface (DHCP or static)
3. ROS2 DDS domain (ros2-domain OpenRC service)  ← first priority
4. Chrony NTP (clock sync for DDS QoS timestamps)
5. Core tier service nodes (dns, dhcp, web, cache, container)
6. High priority tier (vpn, identity, files — if configured)
7. Globe interface (web node + WebSocket bridge)
```

The machine reaches a functional network graph state before it reaches a
usable local state. This is the architectural inversion in practice.

### 3.3 musl libc Compatibility Notes

musl libc differs from glibc in ways relevant to ROS2:

| Issue | Impact | Status |
|---|---|---|
| `dlclose()` is a no-op | Plugin lifecycle must manage memory accumulation | Permanent musl limitation; design around it |
| Thread stack 128 KB (vs 2–10 MB) | Cyclone DDS threads may overflow | Patch required; targeted Cyclone DDS fix |
| No lazy symbol binding | glibc binaries fail on Alpine | Must compile natively on Alpine |
| `__xstat` missing | rcutils test mocking | Fixed upstream (rcutils PR #330) |
| `_NP` pthread extensions | Fast-DDS build failure | Reason Cyclone DDS is mandatory |
| `execinfo.h` absent | Cyclone DDS backtrace | Fixed upstream (cyclonedds PR #384) |
| Python `musllinux` wheel gaps | ROS2 Python tools require source build | Ongoing; contribute wheels upstream |
| DNS parallel queries | DDS discovery non-determinism | Mitigated: use multicast, not DNS |

Full technical assessment: `docs/ROS2_MUSL_COMPATIBILITY.md`

### 3.5 Filesystem Policy

The prosumer incentive model creates direct economic pressure toward storage
efficiency: **every byte consumed locally is a byte that cannot be sold to the
cooperative market**. The default filesystem must therefore maximize usable
capacity, protect data integrity on unexpected power loss, and support
quota-enforced partitioning between cooperative storage and local use.

NTFS is explicitly excluded as a default or native filesystem. It is a
Windows-targeted format requiring the FUSE-based `ntfs-3g` driver on Linux,
which imposes translation overhead and is not suitable for cooperative
infrastructure. NTFS mounts are available read-write only as an interoperability
option when Windows-formatted external media is explicitly attached.

| Node Type | Default Filesystem | Rationale |
|---|---|---|
| x86_64 server / desktop | **Btrfs** (zstd compression, level 3) | CoW, transparent compression, subvolumes, RAID 1 |
| ARM64 / Raspberry Pi (boot) | **F2FS** | Flash-aware wear leveling; extends SD card lifespan |
| ARM64 / Raspberry Pi (data) | **Btrfs** | Same benefits as x86_64; lightweight on Pi 4 |
| Research / NAS node (opt-in) | **ZFS** | Deduplication, ARC cache, large pool management |
| External / USB drives | **Btrfs or ext4** | Portability and broad compatibility |
| Windows media (interop only) | **NTFS (mount-only)** | Read-write via ntfs-3g; never a native format |

**Btrfs** is the primary choice because:
- **Transparent zstd compression** — reduces physical disk usage 30–60% for
  research text, logs, and economic time-series data; the cooperative storage
  market bills on *logical* bytes delivered, so compression is a direct revenue
  multiplier for node operators
- **Subvolumes with quota groups (qgroups)** — hard quota boundaries between:
  `/` (OS, ~2GB), `/var/lib/ntari/` (service state, ~10GB), and
  `/coop-storage/` (cooperative market storage, remainder of disk)
- **Copy-on-Write** — power loss during a write leaves the previous version
  intact; critical for payment database safety (§5.7.4 Layer 3)
- **`btrfs send | receive`** — efficient incremental snapshot streaming; the
  `ntari_backup_node` and `ntari_failover_node` use this for cooperative
  storage backup and CRIU checkpoint transport
- **RAID 1/10 across multiple drives** — multi-disk prosumer nodes get
  redundancy without LVM complexity; managed via `ntari_monitor_node`
- Alpine 3.23 native: `btrfs-progs` in `main` repo; kernel module in-tree

**F2FS** for flash-storage boot volumes because:
- Designed specifically for NAND flash (SD cards, eMMC, NVMe)
- Reduces write amplification that kills SD cards running 24/7 cooperative nodes
- Alpine 3.23 native: `f2fs-tools` in `community` repo

**ZFS** opt-in for research/NAS deployments because:
- **Block-level deduplication** — genomic reference sequences, versioned research
  datasets, and economic archive snapshots frequently contain repeated blocks;
  ZFS dedup can reduce physical storage 40–70%
- **ARC (Adaptive Replacement Cache)** — intelligently caches hot dataset blocks
  in RAM; economic data query performance is exceptional on well-configured pools
- **Trade-offs**: requires ~1GB RAM per 1TB storage; CDDL license (ships via
  Alpine `edge/testing` as `zfs-linux`); too heavy for Raspberry Pi edge nodes
- Enabled only when cooperative governance explicitly selects the NAS role

**Btrfs Subvolume Layout (standard prosumer node):**
```
/dev/sda  (or nvme0n1)
  └─ Btrfs pool
       ├─ @root          mounted at /           quota: 4GB   (OS + logs)
       ├─ @varntari      mounted at /var/lib/ntari/  quota: 20GB  (service state, DBs)
       ├─ @snapshots     mounted at /.snapshots  (Btrfs snapshots, auto-managed)
       └─ @coopstorage   mounted at /coop-storage/  quota: remainder (cooperative market)
```

Quotas enforced by `btrfs qgroup` — the `ntari_monitor_node` publishes quota
usage on `/ntari/monitor/disk_*` topics and alerts when `@coopstorage` exceeds
95% capacity (revenue-impacting threshold).

**Compression billing credit:**
When a node operator sells storage via the cooperative market, billing is on
*logical bytes* (data as seen by the consumer), not physical bytes. If Btrfs
zstd compresses 100GB of research archive to 58GB on disk, the operator is
credited for providing 100GB of cooperative storage while using only 58GB of
physical capacity — a 42% revenue bonus for compression efficiency. The
`ntari_payment_node` reads logical and physical usage from `btrfs fi usage`
and computes the compression credit in each settlement cycle.

## 4. Middleware Layer — ROS2 + Cyclone DDS

### 4.1 Why ROS2

Originally designed for robotics, ROS2's DDS-based communication model provides
exactly the properties needed for community mesh networks:

- **Automatic peer discovery** — no centralized configuration required
- **QoS policies** — survive intermittent connectivity natively
- **Topic-based publish-subscribe** — decouples senders from receivers
- **Queryable computation graph** — makes network topology a visible,
  first-class, human-readable object

The ROS2 computation graph is the DDS graph made legible: a live map of all
nodes, topics, services, and data flows active in the system. The globe
interface renders this graph visually.

### 4.2 Why Cyclone DDS (not Fast-DDS)

Cyclone DDS is the only practical DDS choice for NTARI OS on Alpine:

| Criterion | Cyclone DDS | Fast-DDS |
|---|---|---|
| Implementation language | C (minimal surface) | C++ (Boost, ASIO issues on musl) |
| musl patches | Merged upstream | Require ongoing patching |
| License | EPL-2.0 (no single corporate owner) | Apache 2.0 (eProsima commercial product) |
| Dependency tree | Minimal | ASIO, Boost, OpenSSL (each a musl risk) |
| Alpine build status | Working (SEQSENSE aports) | Problematic |

### 4.3 DDS Topic Schema

#### 4.3.1 System Health Topics

**`/ntari/<service>/health`**
- Publish frequency: 1 Hz
- QoS: Best-effort, volatile
- Type: `bool` (true = healthy)
- Published by: every service node in the services layer

**`/ntari/<service>/status`**
- Publish frequency: 0.1 Hz
- QoS: Reliable, transient-local
- Type: `string` (human-readable status message)
- Published by: every service node in the services layer

**`/ntari/system/heartbeat`**
- Publish frequency: Every 30 seconds
- QoS: Best-effort, volatile

```json
{
  "node_id": "ntari-{uuid}",
  "timestamp": 1739577600,
  "uptime_seconds": 86400,
  "ros2_domain_id": 0,
  "active_nodes": ["ntari_dns_node", "ntari_dhcp_node", "ntari_web_node"],
  "network_quality": {
    "latency_ms": 45,
    "packet_loss": 0.002
  }
}
```

#### 4.3.2 Node Discovery Topics

**`/ntari/network/directory`**
- Publish frequency: Every 60 seconds
- QoS: Reliable, transient-local

```json
{
  "schema_version": "1.5.0",
  "nodes": [
    {
      "node_id": "ntari-{uuid}",
      "ros2_node_name": "string",
      "latency_ms": 45.0,
      "cooperative": "string",
      "federation_remote": false,
      "active_topics": ["string"],
      "last_seen": 1739577600
    }
  ],
  "total_nodes": 12,
  "domain_id": 0
}
```

#### 4.3.3 Governance Topics

**`/ntari/governance/proposals`**
- QoS: Reliable, transient-local (retained)

**`/ntari/governance/votes`**
- QoS: Reliable, volatile

**`/ntari/interface/selected_node`**
- Published by: globe WebSocket bridge when user selects a node
- QoS: Best-effort, volatile

#### 4.3.4 DDS Domain Allocation

```yaml
domains:
  0: local-cooperative    # Default; all nodes in a cooperative share domain 0
  1-9: federation-bridge  # Reserved for DDS domain bridging (inter-coop)
  10-99: application      # Application-specific domains
  100-232: user-defined   # User/deployment-specific
```

### 4.4 WebSocket Bridge Node

The `ntari_globe_bridge` ROS2 node bridges the DDS graph to the browser:

```
ntari_globe_bridge
  ├─ Calls ros2 graph introspection APIs at 1–2 Hz
  ├─ Measures DDS round-trip latency to each peer node
  ├─ Streams node/edge/latency data as JSON over WebSocket
  │     └─ ws://localhost:9090
  └─ Subscribes to /ntari/interface/* topics for globe → DDS feedback
```

**Data format (WebSocket JSON):**

```json
{
  "type": "graph_update",
  "timestamp": 1739577600,
  "nodes": [
    {
      "id": "ntari_dns_node",
      "latency_ms": 2.1,
      "status": "active",
      "topics_pub": ["/ntari/dns/health", "/ntari/dns/status"],
      "topics_sub": []
    }
  ],
  "edges": [
    {
      "from": "ntari_dns_node",
      "to": "ntari_globe_bridge",
      "topic": "/ntari/dns/health"
    }
  ]
}
```

### 4.5 Lifecycle Node Protocol

All NTARI OS service nodes implement the ROS2 lifecycle node protocol:

```
UNCONFIGURED → INACTIVE → ACTIVE → FINALIZED
                  ↑                      ↓
                  └──────────────────────┘
                      (deactivate → cleanup → configure → activate)
```

OpenRC manages the lifecycle transitions as system services, enabling
clean startup and shutdown ordering without systemd dependencies.

## 5. Services Layer

### 5.1 Design Contract

Every service in the services layer must:

1. Run as a ROS2 lifecycle node
2. Be startable/stoppable via OpenRC without affecting other nodes
3. Publish `/ntari/<service>/health` (bool) at 1 Hz
4. Publish `/ntari/<service>/status` (string) at 0.1 Hz
   — **Verbose status is a system standard.** Format (see Appendix 14.5):
   ```
   [HH:MM:SS] <node_name> <STATE> <action_verb> — <key:value key:value ...>
   ```
   - `STATE` must be one of: `ACTIVE | PENDING | COMPLETE | WARN | ERROR | RECOVER`
   - Detail section must contain at minimum one `key:value` pair describing the
     current action. Machine-parseable. Human-readable. No blank status strings.
   - Status messages must be informative enough for a prosumer to diagnose the
     node's condition without reading source code or documentation.
5. Publish `/ntari/<service>/event` (string) **immediately on any state transition**
   — same format as status; fired on-event (not periodic). The Globe event feed
   subscribes to this topic for real-time display. Every WARN, ERROR, RECOVER,
   COMPLETE transition must emit an event regardless of the 0.1 Hz status cycle.
6. Be packaged as an APK in the NTARI APK repository
7. Be licensed AGPL-3.0 or compatible

### 5.7 Payment Settlement & Escrow Node

#### 5.7.1 Rationale

NTARI OS users are **prosumers** — they sell network resources (bandwidth, storage,
compute) to other cooperative members. Payment settlement must therefore be:

1. **Self-hosted** — no third-party custody of cooperative funds
2. **Escrow-capable** — funds held atomically until service delivery confirmed
3. **Tamper-proof** — premature hardware shutdown, process halt, or storage
   deletion cannot cause fund loss (see §5.7.4 Atomic Escrow Safety)
4. **Open source** — auditable by cooperative members; AGPL-3.0 compatible
5. **Cooperative-model fit** — designed for multi-party settlements, not just
   merchant-consumer

#### 5.7.2 Chosen Solution: Three-Layer Payment Stack

After evaluating BTCPay Server, GNU Taler, LNbits, Cyclos Community Edition,
OpenNode, and RoboSats against NTARI OS requirements, the following layered
architecture was selected:

| Layer | System | License | Role | Escrow Mechanism |
|---|---|---|---|---|
| **1 — Lightning** | LNbits | MIT | Lightning micropayments; per-MB and per-vCPU·min billing | HODL invoices (cryptographic hold) |
| **2 — Cooperative Credit** | Cyclos CE | AGPL-3.0 | Fiat/cooperative credit, member accounts, time banking | Native (platform-level escrow transaction type) |
| **3 — Optional E-cash** | GNU Taler | GPL-3.0+ | Privacy-preserving e-cash mint; payer anonymity | Exchange escrow account (deferred; musl validation needed) |

```
NTARI OS Payment Settlement Architecture
=========================================
[ntari_payment_node  — AGPL-3.0 wrapper]
          |
    ┌─────┴──────┐
    |             |
[LNbits]     [Cyclos CE]        ← primary production stack
    |             |
[LND/CLN]   [PostgreSQL]
    |
[Lightning Network]              ← sub-cent micropayments
                    (optional layer 3)
               [GNU Taler]       ← e-cash mint; validate musl before enabling
```

**LNbits** was chosen as the Lightning layer because:
- **HODL invoice escrow** — the closest native Lightning escrow primitive: funds
  are locked in-flight cryptographically until the settlement node reveals the
  preimage, or the invoice expires and funds return automatically. No custom
  plugin required.
- **Multi-wallet architecture** — each cooperative member gets a wallet within
  the shared node; maps naturally to the prosumer model.
- **Clean REST + WebSocket API** — lowest integration complexity for the ROS2
  payment node wrapper; Python SDK (`lnbits-python`) embeddable directly in
  ROS2 Python nodes.
- **Lightest resource footprint** of all candidates (Python/FastAPI, single
  process); important for edge nodes (Raspberry Pi 4).
- **Alpine/Docker compatible** — Python runs natively on Alpine musl.
- MIT license — no conflict with AGPL-3.0.

**Cyclos Community Edition** was chosen as the cooperative credit layer because:
- **Native, full escrow** — escrow is a first-class Cyclos payment type; funds
  are held in a system account pending confirmation, with dispute resolution built
  in. No plugins or workarounds required.
- **AGPL-3.0 license** — perfect match with NTARI OS's own license; network
  service copyleft already satisfied by NTARI OS's AGPL-3.0 requirement.
- **Purpose-built for cooperatives**: mutual credit, time banking, LETS systems,
  demurrage currencies, member account management — all native features.
- **REST API with OAuth2** — integrable from the ROS2 payment node via standard
  HTTP client libraries.
- Runs on JVM (OpenJDK) in a Podman container managed by `ntari_container_node`.

**GNU Taler** remains in the architecture as an optional Layer 3 for deployments
that require payer anonymity (e-cash mint model). Deferred until Alpine/musl libc
compatibility with Taler's C exchange daemon is validated.

#### 5.7.4 Atomic Escrow Safety

**Critical requirement:** User funds must not be losable due to premature hardware
shutdown, process termination, or storage deletion at any point in a transaction.

This is enforced by five independent protection layers:

**Layer 1 — Cryptographic hold (LNbits HODL invoices)**
- A HODL invoice locks funds in the Lightning Network routing path. The funds
  are neither with the sender nor receiver until the preimage is revealed.
- If the `ntari_payment_node` crashes before revealing the preimage, the HODL
  invoice expires automatically (configurable: 300–3600 seconds) and funds
  return to the sender with no loss.
- Hardware shutdown during a HODL cannot cause loss: funds are in the Lightning
  Network's cryptographic hold, not on disk. No local state is needed to trigger
  the automatic refund on expiry.

**Layer 2 — Cyclos ACID escrow (cooperative credit transactions)**
- Cyclos escrow transactions are persisted to PostgreSQL with full ACID guarantees.
- Funds in a Cyclos escrow system account are not spendable by any party until
  the escrow is explicitly released (delivery confirmed) or refunded (disputed).
- Power loss mid-escrow leaves funds in the Cyclos system escrow account —
  neither transferred to seller nor returned to buyer — until a recovery action
  is taken on next boot.

**Layer 3 — ROS2 lifecycle state machine**
- `ntari_payment_node` uses the ROS2 managed lifecycle:
  `UNCONFIGURED → INACTIVE → ACTIVE → FINALIZATION`
- The node will not transition from `ACTIVE` to `SHUTDOWN` while:
  - any Cyclos escrow is in `PENDING` state, OR
  - any HODL invoice is unrevealed and unexpired.
- OpenRC `stop` blocks on the lifecycle state (configurable timeout: 300 seconds).
- On timeout, the node writes an `escrow_recovery.json` checkpoint and shuts down.
  Recovery runs automatically on next boot before `ntari_payment_node` activates.

**Layer 4 — Storage deletion protection**
- Payment database directories are protected by AppArmor:
  - `/var/lib/ntari/payment/lnbits/` — writable only by `lnbits` process
  - `/var/lib/ntari/payment/cyclos/` — writable only by `postgres` process
- Both AppArmor profiles deny `unlink()` and `rename()` to all other processes.
- Emergency deletion of payment records requires cooperative governance quorum
  to override AppArmor policy — no single admin can unilaterally destroy records.

**Layer 5 — Watchdog + Globe alert**
- `ntari_payment_node` publishes `/ntari/payment/health` at 1 Hz.
- Absent heartbeat for 5 seconds → watchdog publishes to `/ntari/system/alerts`.
- Globe interface shows: red node pulse + escrow count badge + amber admin banner
  "Payment node unreachable — X escrow contracts at risk."
- Cooperative members online see the alert simultaneously via Globe WebSocket.

#### 5.7.5 Prosumer Billing Model

Resources priced in **NTARI_CREDITS** (internal cooperative unit):

| Resource | Billing Unit | Settlement Layer | Escrow Type |
|---|---|---|---|
| Bandwidth (upload) | per MB | Lightning (LNbits HODL) | Cryptographic hold |
| Bandwidth (download) | per MB | Lightning (LNbits HODL) | Cryptographic hold |
| Storage hosting | per GB·hour | Cyclos CE escrow | Platform escrow account |
| Compute sharing | per vCPU·min | Lightning (LNbits HODL) | Cryptographic hold |
| VPN relay | per connection·hour | Cyclos CE escrow | Platform escrow account |
| DNS/DHCP service | cooperative flat fee | Cyclos CE batch | Member account debit |
| Inter-cooperative | per service agreement | Cyclos CE escrow | Platform escrow account |

1 NTARI_CREDIT = 1 Cyclos credit unit (exchange rate set by cooperative governance).
Lightning micropayments denominated in satoshis; converted at settlement close.

#### 5.7.6 Implementation Notes

- **LNbits** requires: Python 3.11+, `py3-fastapi`, `py3-uvicorn`, `py3-httpx`
  — all available in Alpine 3.23. Connects to LND or Core Lightning (CLN).
  CLN recommended (better Alpine/musl compatibility than LND's Go runtime).
- **Cyclos CE** runs as a JVM application (OpenJDK 21) inside a Podman container
  managed by `ntari_container_node`. Official Cyclos Docker images are Debian-based
  but run cleanly on Alpine-hosted Docker/Podman without modification.
- **PostgreSQL 16** required by Cyclos CE — add `postgresql16` to APK list;
  already required by FreeIPA identity node.
- **Exchange private signing keys** (LNbits macaroon, Cyclos admin credentials)
  stored in LUKS-encrypted volume mounted at boot; keys never touch unencrypted disk.
- **GNU Taler** (Layer 3, optional): Requires validation of C exchange daemon
  (`taler-exchange`) against Alpine musl libc before enabling. Add to v1.6 scope
  if musl compatibility is confirmed.

#### 5.9.1 Design Goal

A **highly accessible, network-aware clipboard** that makes cross-node copy/paste

multiple nodes — running terminal sessions through the bastion, monitoring the
globe, and managing payment contracts — should be able to copy from one node and

**Accessibility emphasis**

### 5.10 Failover & Process Migration Node (`ntari_failover_node`)

#### 5.10.1 Design Goal

For active resource transmissions — CPU/GPU compute, RAM leasing, network relay,
live data streams — a "backup" is not a static file copy. It is a **hot standby
process on a second cooperative node** that can resume the work within seconds of
primary failure. The `ntari_failover_node` orchestrates this, funded automatically
by each transaction's fee structure (§5.7.5).

This creates a **cooperative standby market**: idle nodes with good connectivity
earn NTARI_CREDITS simply by being a reliable hot standby for active nodes — they
earn without serving. For the prosumer, this means guaranteed continuity; for
idle-capacity holders, it means passive income.

#### 5.10.2 CRIU — Checkpoint/Restore In Userspace

**CRIU** (Apache-2.0 license, available in Alpine 3.23 `community` repo) is the
core technology enabling process migration:

- Snapshots a running Linux process tree to a set of image files without stopping it
- Snapshot includes: memory pages, file descriptors, sockets, pipe state, CPU registers
- The snapshot can be transferred to another node and resumed there — the process
  continues from the exact point it was checkpointed
- Used in production by Google (live migration), CERN (HPC checkpoint/restart),
  and container runtimes (Podman checkpoint/restore)

**CRIU on Alpine musl**: CRIU is C-based and compiles cleanly against musl libc.
Alpine 3.23 ships `criu` in the `community` repository.

#### 5.10.3 Standby Market SLA Tiers

Prosumers select a failover SLA tier when purchasing active resources:

| Tier | Switchover Time | Standby Model | CRIU Mode | Fee Premium |
|---|---|---|---|---|
| **Best Effort** | Minutes | Cold (checkpoint on disk, not loaded) | `dump` only | +5% |
| **Standard** | < 30 seconds | Warm (checkpoint in RAM, pre-loaded) | `dump` + pre-restore | +15% |
| **Hot** | < 5 seconds | Hot (process paused on standby, ready to resume) | `pre-dump` loop | +35% |
| **Mirror** | < 500ms | Active-active (dual primary, DDS-synchronized state) | Live migration | +80% |

Standby nodes bid for standby slots by publishing their LBTAS score and available
capacity to `/ntari/failover/standby_capacity`. The payment node selects the
highest-LBTAS-rated standby that meets the SLA tier requirement.

#### 5.10.4 Failover by Resource Type

**CPU/GPU compute jobs**
```
1. ntari_failover_node initiates CRIU pre-dump on primary node (memory dirty tracking)
2. Checkpoint delta streams to standby node via WireGuard at configurable interval
3. On heartbeat loss (primary silent >2s): standby resumes from last checkpoint
4. Client receives switchover notification on /ntari/failover/switchover_event
5. ntari_payment_node pro-rates primary payment; activates standby as new primary
```

**RAM leasing**
```
1. Primary leased memory pages are shadowed to standby node via DDS-published deltas
2. On primary failure: standby promotes shadow copy to active — client remaps pointer
3. Continuity: no data loss beyond last shadow sync interval (configurable: 100ms–5s)
```

**Network relay (WireGuard)**
```
1. WireGuard standby peer holds pre-negotiated keys (shared during session setup)
2. On primary relay failure: Cyclone DDS multicast detects topology change
3. ntari_failover_node broadcasts reroute; clients reconnect to standby peer
4. DDS domain bridge re-establishes through standby — typically transparent to consumers
```

**Cooperative storage (Btrfs)**
```
1. ntari_backup_node streams btrfs snapshots to standby node via btrfs send | receive
2. On primary failure: standby node promotes snapshot to live; serves from last snapshot
3. Billing gap: consumer is not charged for the snapshot lag window
4. ntari_lbtas_node auto-rates primary node down for the continuity gap
```

**Active data streams (research pipelines, economic feeds)**
```
1. Standby node subscribes as secondary consumer, buffering last N seconds
2. On primary failure: standby promotes itself; replays buffer to all consumers
3. Buffer size is SLA-tier-dependent: Best Effort=30s, Standard=120s, Hot=300s
```

#### 5.10.5 Fee Structure Integration

Every resource transaction fee automatically includes standby cost:

```
Total escrow = primary_fee + standby_fee + switchover_sla_bond

switchover_sla_bond = fee_penalty paid FROM standby TO consumer IF SLA is missed
```

The `switchover_sla_bond` is held in a separate Cyclos escrow sub-account. If
the standby node fails to activate within the contracted SLA window, the bond is
released to the consumer as a service credit. This makes SLA commitments credible
— the standby node has skin in the game.

### 5.11 LBTAS Node (`ntari_lbtas_node`)

#### 5.11.1 What LBTAS Is

The **Leveson-Based Trade Assessment Scale** (LBTAS) is NTARI's native reputation
system, developed by the Network Theory Applied Research Institute. It applies
Nancy Leveson's aircraft software safety assessment methodology — designed for
contexts where failures cost lives — to cooperative digital commerce.

The 5-star rating system (1958, Forbes Travel Guide) was built for one-way
highway hotel advertising. LBTAS replaces it with a system built for
**bidirectional, data-dense, producer-accountable cooperative transactions**.

Source repository: `https://github.com/NTARI-RAND/Leveson-Based-Trade-Assessment-Scale`
License: AGPL-3.0 (perfect match with NTARI OS)

#### 5.11.2 The Scale

LBTAS uses a **6-point scale from -1 to +4**:

| Score | Label | Definition |
|---|---|---|
| **+4** | Delight | Interaction anticipates the evolution of user practices and concerns post-transaction |
| **+3** | No Negative Consequences | Interaction designed to prevent loss, exceed basic quality standards |
| **+2** | Basic Satisfaction | Interaction meets socially acceptable standards, exceeding articulated user demands |
| **+1** | Basic Promise | Interaction meets all articulated user demands, no more |
| **0** | Cynical Satisfaction | Interaction fulfills a basic promise requiring little to no discipline toward user satisfaction |
| **-1** | No Trust | User was harmed, exploited, or received a product/service with evidence of no discipline or malicious intent |

The scale compresses meaning into each level — a +3 is not "pretty good," it is a
specific claim: no negative consequences occurred and basic quality was exceeded.
This precision enables the system to drive improvement without relying on comment
sections.

#### 5.11.3 Bidirectional Assessment

Every NTARI OS transaction generates **two ratings**:
- **Producer rating** — the node providing the resource is rated by the consumer
- **Consumer rating** — the node receiving the resource is rated by the producer

This bidirectionality is fundamental to cooperative governance:
- A consumer who repeatedly disputes legitimate completions accumulates poor
  consumer ratings — making standby nodes reluctant to serve them
- A producer who delivers at +4 consistently commands premium LBTAS multipliers
- Neither party can game the system unilaterally; both are accountable

#### 5.11.4 Rating Categories (NTARI OS context)

The LBTAS default categories (reliability, usability, performance, support) are
extended with NTARI OS-specific categories for automated system rating:

| Category | Rated By | Automation Source |
|---|---|---|
| **reliability** | auto + peer | Heartbeat gaps from `ntari_monitor_node`; uptime % |
| **performance** | auto + peer | Actual vs. contracted SLA tier (§5.10.3) |
| **integrity** | auto | CRIU checkpoint completeness; Btrfs checksum verification |
| **continuity** | auto | Switchover time; buffer gap on stream failover |
| **settlement** | auto | LNbits/Cyclos settlement success rate; escrow dispute rate |
| **usability** | peer only | Peer-submitted rating; Globe UI interaction |
| **support** | peer only | Governance dispute resolution quality; response time |

Automated categories are rated without human input — the `ntari_lbtas_node`
reads telemetry from `ntari_monitor_node`, `ntari_failover_node`, and
`ntari_payment_node` and submits ratings programmatically at transaction close.

#### 5.11.5 LBTAS Score → Pricing Multiplier

A node's aggregate LBTAS score (weighted average across all categories, all
transactions) maps directly to the rate it can charge in the cooperative market:

| Aggregate Score | Label | Pricing Multiplier | SLA Tier Eligibility |
|---|---|---|---|
| 3.5 – 4.0 | Delight-tier | ×1.30 | All tiers including Mirror |
| 2.5 – 3.4 | No-Negative-tier | ×1.15 | Hot and below |
| 1.5 – 2.4 | Satisfaction-tier | ×1.00 (standard rate) | Standard and below |
| 0.5 – 1.4 | Promise-tier | ×0.80 | Best Effort only |
| -0.4 – 0.4 | Cynical-tier | ×0.55 | Best Effort only; flagged |
| < -0.4 | No-Trust | Suspended from market | Governance review required |

A new node with no transaction history starts at the Satisfaction-tier (×1.00,
Standard SLA) — it can participate immediately but cannot command premiums until
it has built a rated history. This prevents exclusion of new cooperative members
while maintaining incentives for proven nodes.

**Score weighting by recency**: The most recent 30 days of transactions carry
3× the weight of older transactions — LBTAS rewards improvement, not just legacy.

#### 5.11.6 ROS2 Node Specification

The `ntari_lbtas_node` wraps the LBTAS Go implementation (single binary, fast

#### 5.11.7 Federation and Privacy

- LBTAS scores are **public within a cooperative** — all members can query any
  node's score via the Globe interface; this is intentional (community self-regulation)
- **Inter-cooperative**: scores are shared across WireGuard federation links under
  a federation agreement; receiving cooperatives can choose to trust, weight, or
  ignore scores from federated peers
- **Consumer privacy**: consumer ratings are aggregated — no individual consumer
  rating is exposed to the producer; only the aggregate changes. This prevents
  retaliation.
- **Dispute resolution**: a disputed rating (producer claims consumer rated in bad
  faith) goes to the governance system (§10); quorum vote can expunge a rating via
  `/ntari/lbtas/governance_override`

### 6.1 Design Premise

Every mainstream OS presents a local-first metaphor: a desktop, a file system,
a launcher — implying the machine is the primary object of concern.

NTARI OS inverts this. The network is the primary object. The machine is one
participant in a larger cooperative graph. The interface expresses this
inversion immediately and viscerally. The first thing a user encounters is
the network they belong to.

### 6.2 The Globe as Instrument

The globe is not a geographic map. It carries no political borders, no
coastlines, no terrestrial features. It is an abstract sphere — a scientific
instrument, like an orrery — whose only purpose is to express topological
relationships between network nodes.

- **No geography prevents misreading.** Node position represents network
  distance from the local node, not physical location.
- **Abstract sphere is politically neutral.** Cooperative infrastructure must
  not privilege any region visually.
- **The instrument aesthetic communicates precision** — a wireframe lattice
  sphere communicates "built by people who care about accuracy."

### 6.3 Globe Behavior Specification

**Radius scaling:**
The globe's radius grows logarithmically with node count.

| Nodes | Globe behavior |
|---|---|
| 0–2 | Small, intimate. Generous space around it. |
| 3–8 | Noticeably larger. The network has presence. |
| 9–15 | Fills substantial viewport. Auto-zoom engages. |
| 16–30 | Near-maximum radius. Auto-zoom compresses significantly. |
| 30+ | Maximum radius. Zoom compresses further. |

**Node placement:**
- Node angular distance from local node ∝ network latency
- Highest-latency node is always diametrically opposite local node
- New nodes animate in over ~40 frames

**Auto-zoom:**
- Engages when >10 nodes visible in display area
- Compresses smoothly and continuously — no snapping
- Manual zoom overrides for 3 seconds, then returns to auto

**Node visual treatment:**
- Default: crisp white dot (2–4px radius) with slow-pulsing outer glow
- Each node has a unique pulse phase and frequency — constellation feels alive
- Back-hemisphere nodes: ~12% opacity (visible but clearly behind)
- Selected: warm orange (`#ff6b35`) with outer ring
- Hover: pointer cursor, slight radius increase + performance overlay panel (see §6.7)

**Edges:**
- Faint quadratic bezier curves (not straight lines)
- Maximum 25% opacity — nodes carry primary meaning
- Fade in over ~1 second after appearing
- Back-hemisphere edges: further reduced opacity; cross-hemisphere edges fade mid-arc

**Labels:**
- JetBrains Mono, 10px
- Right-aligned adjacent to node dot
- Fade in during second half of node appearance animation
- Hidden for back-hemisphere nodes
- Reduced opacity when node not selected/hovered
- Recommended naming: `<domain>.<function>` (e.g., `core.scheduler`, `mesh.relay_0`)

**Search:**
- Positioned right-center of viewport
- Searches by node name and topic name
- Selection orients globe toward selected node
- Node detail panel shows: status, latency_ms, RMW, subscribed topics

### 5.12 Snapshot Node (`ntari_snapshot_node`)

#### 5.12.1 Design Goal

CompTIA-documented best practice: a system should be restorable to a previous
point in time when installing an app, driver, or update causes improper function.
`ntari_snapshot_node` implements this as a first-class, automatic, user-visible
feature — not a sysadmin tool requiring manual intervention.

Btrfs copy-on-write snapshots (§3.5) are the underlying mechanism. The snapshot
node makes them automatic, named, retention-managed, Globe-visible, and
cooperative-network-recoverable.

#### 5.12.2 Automatic Snapshot Triggers

A snapshot of `@root` and `@varntari` subvolumes is taken automatically before
every potentially destructive operation:

| Trigger | Who fires it | Snapshot name suffix |
|---|---|---|
| `apk add` / `apk del` | `ntari-apk` wrapper (pre-install hook) | `_apk_install_<pkgname>` |
| `apk upgrade` | `ntari-apk` wrapper | `_apk_upgrade` |
| ROS2 node APK upgrade | `ntari-apk` wrapper | `_node_upgrade_<nodename>` |
| Service config write | `ntari_<service>_node` on config change | `_config_<servicename>` |
| WireGuard config change | `ntari_vpn_node` | `_vpn_config` |
| Governance vote execution | `ntari_governance_node` | `_governance_<vote_id>` |
| Manual admin trigger | `ntari-snapshot create <label>` CLI | `_manual_<label>` |
| Pre-scheduled (daily) | OpenRC cron via `ntari_snapshot_node` | `_scheduled_daily` |

Snapshot naming format:
```
@snap_YYYYMMDD_HHMMSS<suffix>
Example: @snap_20260218_143022_apk_upgrade
```

#### 5.12.3 Snapshot Retention Policy

Default retention (configurable via governance):

| Retention class | Count kept | Age threshold |
|---|---|---|
| Pre-operation snapshots | 10 most recent | Auto-pruned when 11th created |
| Scheduled daily | 7 | Older than 7 days pruned |
| Scheduled weekly | 4 | Older than 4 weeks pruned |
| Manual / labelled | Unlimited | Never auto-pruned; require manual delete |
| Pre-governance | All | Retained until vote outcome confirmed |

Pruning runs after each new snapshot is created. `ntari_snapshot_node` publishes
a `COMPLETE prune` event with a count of snapshots removed and bytes reclaimed.

#### 5.12.4 Boot-Time Rollback

If a node fails to reach the `ACTIVE` state within the boot sequence timeout,
the initramfs boot menu automatically presents a snapshot restore option:

```
NTARI OS Boot Recovery
══════════════════════════════════════════════════════
  Last successful boot: 2026-02-18 09:12:04

  Available restore points:
  [1] @snap_20260218_143022_apk_upgrade    (14 min ago, 847MB)
  [2] @snap_20260218_091204_scheduled_daily (6 hr ago,  851MB)
  [3] @snap_20260217_091156_scheduled_daily (1 day ago, 849MB)

  [S] Drop to recovery shell
```

Implementation:
- GRUB is configured to boot from any named Btrfs subvolume
- initramfs script reads `/.snapshots/` and generates the menu
- Restore = GRUB default-subvolume repoint + reboot (< 30 seconds total)
- The broken subvolume is not deleted — it is renamed `@root_failed_YYYYMMDD`
  and kept for 7 days for diagnostic purposes

#### 5.12.5 Cooperative Snapshot Witness

For physical disk failure (not just software breakage), local snapshots are
insufficient. `ntari_snapshot_node` optionally ships OS snapshots to a designated
**witness node** on the cooperative network:

```
Local node                        Witness node
@snap_YYYYMMDD  ──btrfs send──▶  received + stored in /coop-storage/snapshots/<nodeid>/
                   WireGuard
```

- Transport: `btrfs send | btrfs receive` piped over WireGuard tunnel
- Frequency: daily scheduled snapshot is sent; pre-operation snapshots are not
  (too frequent; too much bandwidth)
- Cost: snapshot storage on witness node is billed to the owner node at the
  standard cooperative storage rate (per GB·hour via Cyclos CE escrow)
- Restore: on hardware failure, new node requests snapshot from witness via
  `ntari-restore --from-witness <witness_node_id> <snap_name>`; the witness
  streams it back; new node boots from it

### 5.13 Witness Node (`ntari_witness_node`)

#### 5.13.1 Design Goal

Witness nodes serve two revenue streams from a single trusted role:

1. **Snapshot hosting** — store OS snapshots for other cooperative nodes;
   earn storage credits (§5.12.5)
2. **LBTAS dispute arbitration** — review and adjudicate disputed transactions;
   earn arbitration fees

Both revenue streams require the same underlying trust profile: high LBTAS score,
reliable uptime, and demonstrated cooperative accountability. A node that qualifies
for one qualifies for both.

**Minimum requirements to register as a witness node:**
- LBTAS aggregate score ≥ 2.5 (No-Negative-tier or above)
- Uptime reliability ≥ 95% (rolling 30-day)
- Available storage ≥ 50GB dedicated to `@coopstorage`
- Node has been active in the cooperative for ≥ 30 days

#### 5.13.2 Dispute Lifecycle

Every LBTAS dispute follows a defined lifecycle:

```
FILED → QUEUED → CLAIMED → EVIDENCE → DELIBERATION → DECIDED → [APPEAL] → CLOSED
```

**FILED**: A party submits a dispute via `/ntari/lbtas/dispute_file`:
- Triggered automatically when a rating of `-1` is submitted (No Trust)
- OR manually by either party contesting any rating within 72 hours
- Filing deducts the dispute fee from the filer's Cyclos account into escrow

**QUEUED**: Dispute enters the witness queue, visible to all eligible witness nodes.
Sorted by: stake value (desc), time waiting (asc), SLA tier of original transaction.

**CLAIMED**: A witness node operator claims the dispute via the dashboard.
Claim locks the dispute to that node for 24 hours. If unclaimed after 48 hours,
the fee increases by 20% (incentivizes faster claiming) and is re-broadcast.

**EVIDENCE**: System auto-collects and presents:
- Full telemetry from `ntari_monitor_node` for the transaction window
- Checkpoint records from `ntari_failover_node` (was SLA met?)
- Settlement records from `ntari_payment_node` (did payment complete?)
- Status event log from `/var/log/ntari/events.log` for both nodes
- Both parties may submit a written statement (max 500 words each, 24h window)

**DELIBERATION**: Arbitrator reviews evidence. No time limit, but:
- LBTAS arbitration quality score tracks time-to-decision (target: < 8 hours)
- Long deliberations reduce arbitration quality score

**DECIDED**: Arbitrator selects outcome:
- **Uphold** — dispute filer was correct; respondent's rating stands or is applied
- **Dismiss** — dispute was without merit; filer's rating is reversed; filer pays fee
- **Partial** — split outcome; fee split proportionally
- Required: minimum 100-word written rationale, stored permanently to cooperative ledger

**APPEAL**: Either party may appeal once, within 48 hours of decision.
Appeal escalates to a panel of 3 witness nodes (selected by highest LBTAS score
among available witnesses). Appeal fee = 3× base dispute fee. Panel majority decides.

**CLOSED**: Final outcome applied:
- LBTAS scores updated via `/ntari/lbtas/governance_override`
- Escrow fee released to arbitrator(s)
- If filer lost: filer's dispute fee is non-refundable; respondent receives courtesy credit
- All records sealed to cooperative ledger (immutable, queryable by governance)

#### 5.13.3 Dispute Fee Budget

**Phase 1 — Bootstrap (< 90 days of dispute history):**

Governance-set initial values:
```
floor_fee          = 0.5 NTARI_CREDITS   (minimum fee; deters frivolous micro-disputes)
dispute_percentage = 2.0%                (of disputed transaction value)
dispute_fee        = max(floor_fee, dispute_percentage × transaction_value)
```

The floor prevents griefing (filing disputes on negligible transactions to harass
nodes). The percentage scales with stakes.

**Phase 2 — Historical calibration (≥ 90 days of dispute history):**

`ntari_lbtas_node` computes a rolling 90-day dataset:
```
dispute_rate         = disputes / total_transactions
avg_resolution_hrs   = mean hours from FILED to DECIDED
arbitrator_cost      = avg_resolution_hrs × arbitrator_credit_rate
                       (arbitrator_credit_rate set by governance; initially 1.0 cr/hr)
frivolous_rate       = disputes_decided_against_filer / total_disputes
frivolous_penalty    = frivolous_rate × 2.0   (penalty multiplier on fee)
```

Calibrated fee formula:
```
dispute_fee = max(floor_fee, arbitrator_cost × (1 + frivolous_penalty))
```

If 20% of disputes are frivolous: penalty = 0.40 → fee = arbitrator_cost × 1.40.
This deters bad-faith filing while keeping legitimate disputes affordable.

Fee parameters are recalculated each governance cycle (default: monthly) and
published to `/ntari/witness/fee_schedule`. Any node can query current fees.

**System budget reserve:**
A per-transaction levy of 0.1% of all settled transactions accumulates in a
Cyclos CE system account (`ntari-dispute-reserve`). When a dispute is upheld
(filer was right), the arbitration fee is drawn from this reserve — the injured
party pays nothing to seek redress. The reserve balance is published publicly on
`/ntari/witness/reserve_balance` and visible in the Globe Admin → Payment tab.

### 7.1 Inter-Cooperative Federation

```
Cooperative A                        Cooperative B
┌────────────────┐                  ┌────────────────┐
│ DDS Domain 0   │                  │ DDS Domain 0   │
│ (local graph)  │                  │ (local graph)  │
│                │  WireGuard       │                │
│ ntari_vpn_node ├──────────────────┤ ntari_vpn_node │
│                │  tunnel          │                │
│ DDS Bridge     ├──────────────────┤ DDS Bridge     │
│ (remote.*)     │                  │ (remote.*)     │
└────────────────┘                  └────────────────┘
```

**How it works:**
1. WireGuard establishes an encrypted tunnel between cooperatives (51820/UDP)
2. Cyclone DDS domain bridge propagates the local graph across the tunnel
3. Remote nodes appear in the local DDS domain with `remote.` prefix
4. Remote nodes are visible in the globe interface with federation-blue visual
5. Federation is governed by a signed YAML federation agreement file

### 7.2 Federation Agreement Format

```yaml
# /etc/ntari/federation/agreements/coop-b.yaml
federation_version: "1.5"
cooperative_id: "coop-b.example.org"
wireguard_endpoint: "203.0.113.1:51820"
wireguard_public_key: "base64-encoded-key"
allowed_domains: [0]
remote_prefix: "remote.coop-b"
agreed_at: "2026-02-17T00:00:00Z"
signed_by: "admin@coop-a.example.org"
signature: "ed25519:base64-signature"
```

### 7.3 Comparison to v1.0 Approach

v1.0 proposed libp2p + Kademlia DHT. v1.5 replaces this with WireGuard + DDS
domain bridging because:

- WireGuard is simpler (fewer moving parts), more auditable, in-kernel
- DDS domain bridging is a native Cyclone DDS capability — no extra protocol
- Federation is visible in the same graph model as local nodes
- Administrators can understand and audit WireGuard tunnels; libp2p DHT
  is significantly harder to reason about

libp2p may still be evaluated for bootstrap/discovery use cases in future.

## 8. Security & Privacy Framework

### 8.1 Defense in Depth

1. **Network Layer:** iptables firewall, restrictive defaults
2. **Application Layer:** Service hardening, AppArmor profiles per node
3. **File System:** AIDE integrity monitoring
4. **Access Control:** MAC with AppArmor
5. **Intrusion Detection:** fail2ban
6. **Identity:** FreeIPA (Kerberos-based) for cooperative members
7. **Channels:** WireGuard (all inter-cooperative traffic)

### 8.2 Firewall Defaults

```
INPUT: DROP (default)
  ├─ ACCEPT: loopback
  ├─ ACCEPT: established, related
  ├─ ACCEPT: SSH (22/TCP)
  ├─ ACCEPT: DDS multicast (239.255.0.1:7400-7500/UDP — local network only)
  ├─ ACCEPT: WireGuard (51820/UDP — when vpn node active)
  ├─ ACCEPT: Globe interface (8080/TCP — localhost only by default)
  ├─ ACCEPT: RADIUS Auth (1812/UDP — when radius node active)
  ├─ ACCEPT: RADIUS Acct (1813/UDP — when radius node active)
  └─ ACCEPT: ICMP echo-request

FORWARD: DROP (default)
OUTPUT: ACCEPT (default)
```

## 10. Governance System

### 10.1 Governance Principles

Governance in v1.5 is community-driven and enforced by the AGPL-3.0 license:

- **AGPL-3.0 requires** that modifications to network-deployed services be
  shared with the network. This is not optional — it is structural.
- **Modular node packages** — each service is independently auditable and
  replaceable via community vote
- **NTARI APK repository** — community-governed package acceptance; new
  service nodes require community review before inclusion

### 10.2 Proposal Lifecycle

```
DRAFT → ACTIVE → PASSED/REJECTED → IMPLEMENTED
```

**Scopes and quorum:**

| Scope | Quorum Required |
|---|---|
| Local (single cooperative) | 50% of local nodes |
| WAN (multi-cooperative) | 25% of participating nodes |
| Global (all network) | 20% of all nodes |

**Process:**
1. Any node with uptime >7 days may submit a proposal
2. 24-hour review period (DRAFT state)
3. 7-day voting period (ACTIVE state)
4. Quorum check at close; PASSED or REJECTED
5. Passed proposals implemented in next release cycle

### 10.3 Governance Topics (DDS)

```
/ntari/governance/proposals    # Active and historical proposals
/ntari/governance/votes        # Individual vote records (anonymous by default)
/ntari/governance/results      # Final tally and outcome
```

### 10.4 Service Layer Governance

Community tier services may be:
- **Added**: Proposed, reviewed, and accepted into the NTARI APK repo
- **Replaced**: Alternative implementation proposed (e.g., different chat server)
- **Removed**: Deprecated and removed from default install

All changes require following the proposal lifecycle above.

### 14.5 Status Message Format Reference

**Verbose status messages are a system standard (§5.1).** This appendix is the
complete reference for all defined STATE values, ACTION verbs, and required
DETAIL fields per node. Implementations must conform to this reference.

#### Format

```
[HH:MM:SS] <node_name> <STATE> <action_verb> — <key:value key:value ...>
```

All fields are required. No field may be empty. Detail section requires at
minimum one `key:value` pair. Keys and values are lowercase, snake_case.
Multiple values use comma separation within a single key: `ifaces:eth0,wg0`.

#### STATE Values

| STATE | Meaning | Globe indicator |
|---|---|---|
| `ACTIVE` | Normal operation; action is in progress | None (expected baseline) |
| `PENDING` | Waiting on an external condition or party | Amber dot on node |
| `COMPLETE` | Action finished successfully | Brief green flash on node |
| `WARN` | Degraded but still functional; no intervention required yet | Amber node tint |
| `ERROR` | Action failed; may require intervention | Red node pulse |
| `RECOVER` | Automatic recovery in progress | Cyan node pulse |
